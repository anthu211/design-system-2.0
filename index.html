A design system is a collection of reusable components, guided by clear standards, that can be assembled to build any number of applications. It bridges design and development teams with a shared language.

This design system uses CSS custom properties (variables) with a body.theme-light class toggle. Dark mode is the default. Toggle the sun/moon icon in the top bar to switch themes. Preference is persisted via localStorage.

All modern browsers: Chrome 90+, Firefox 88+, Safari 14+, Edge 90+. Internet Explorer is not supported. The design system uses CSS Grid, Flexbox, CSS Custom Properties, and SVG.

States

Canonical patterns for loading, empty, and error states across every component type. Pick the right pattern — never invent a new one.

When to use which
Loading
Data is being fetched. Show skeleton instead of spinner wherever layout is known.
Empty
Query returned 0 results, or no data has been added yet. Never hide table headers.
Error
Request failed or page cannot render. Scope error to the failing section — don't kill the whole page.
Loading — Skeleton

Use skeleton loaders whenever the layout is predictable. They reduce perceived wait time better than a spinner.

Error Handling

5 error contexts — choose the right pattern for each

The correct pattern depends on how much of the page is affected and whether the user can take recovery action. Full-page system errors use heavy visual treatment. Section errors contain the problem. Empty states are not errors — they guide the user's next step.

Pattern 1 — Full-Page System Error
When: entire page cannot render (server error, auth failure, network down)
Full page
ERROR
🚧
Oops! That Wasn't Supposed to Happen
Well, this is awkward.. Something broke on our end. We're fixing it ASAP!
✅ Rules
  • "ERROR" watermark: 90px+, weight 800, opacity 8%, #a3a5af, centred behind content
  • 🚧 construction emoji, 40px, centred — signals active incident
  • Heading: friendly, non-technical, SemiBold 18px, #101010
  • Subtext: conversational, 13px, #a3a5af — not a stack trace
  • Refresh button: #6360d8, border-radius: 4px, height 32px
  • Shell (topbar + nav) remains fully visible — users need an escape route
❌ Never
  • Never expose raw stack traces, HTTP codes, or error IDs to the user
  • Never use "500 Internal Server Error" as the visible heading
  • Never remove the navigation shell — the user needs to navigate away
  • Never use 🚦 (traffic light) for system errors — that's for empty states
  • Never auto-redirect without first displaying this state
Pattern 2 — Section / Data Retrieval Error
When: one widget or section fails to load; the rest of the page works
Section
🚧
Data Retrieval Failed
We're having trouble loading this data. Try refreshing or check back later.
✅ Rules
  • No "ERROR" watermark — this is a contained section failure
  • 🚧 emoji, smaller (30px) — same signal, lower visual weight
  • Heading names the data that failed ("Data Retrieval Failed")
  • Include a Refresh button — section data can be retried independently
  • All other page sections remain fully functional
❌ Never
  • Never redirect the entire page because one section failed
  • Never use the full-page "ERROR" watermark treatment here
  • Never omit the Refresh button — section errors are usually retryable
  • Never collapse or hide the widget container — keep it visible
Pattern 3 — Table Empty State (No Data)
When: query returns 0 rows — filters too restrictive, or data not yet ingested
Empty state
🚦
No Data… For Now!
Try adjusting your filters or check back later.
✅ Rules
  • 🚦 traffic light emoji — signals "wait, not broken"
  • "No Data… For Now!" — friendly, reassuring, not alarming
  • Subtext guides next user action ("adjust filters")
  • No Refresh button — this is a user-driven state
  • Table header row and pagination remain visible
❌ Never
  • Never show bare "No records found." with no guidance
  • Never use 🚧 for empty state — that signals a system error
  • Never hide the table header when in empty state
  • Never show a Refresh button for filter-driven empty state
Pattern 4 — Inline Field Validation Error
When: a form field fails validation on blur or on submit
Form
API key is required
Must be a number between 0 and 10
✅ Rules
  • Red border #dc2626, 1.5px + inline ⓘ icon + message text
  • Error message: specific and actionable (not "Invalid input")
  • Validate on blur (not every keystroke) to avoid disrupting flow
  • On submit, scroll to and focus the first invalid field
  • Keep the user's entered value — never clear on error
❌ Never
  • Never show a red border alone — always pair with an error message
  • Never write "Invalid" or "Error" without explaining what to fix
  • Never validate on every keystroke — only on blur and submit
  • Never clear the field value on an error state
Pattern 5 — Toast Notifications
Transient feedback — auto-dismiss after 3s for success; error toasts persist
Transient

✅ Rules
  • Success and info toasts: auto-dismiss after 3 seconds

  • Stack up to 3 toasts max — queue any additional
  • Always include a ✕ close button
  • Message is specific — "1,240 records imported", not "Done"
❌ Never
  • Never use toast for critical errors requiring a user decision — use modal
  • Never auto-dismiss error toasts — user must acknowledge
  • Never stack more than 3 toasts at once
  • Never position toasts top-left or top-centre
  • Never use vague messages like "Done" or "Error occurred"

Documentation

Non-negotiable dos and don'ts for every screen built on this design system.

Layout & Shell

✅ Do
  • Use the page shell verbatim — topbar, left nav, sub-header, content slot only.
  • Topbar background is always #131313 — use PAI logo image only, never text.
  • Minimum supported resolution: 1024px. Primary target: 1440px.
  • Sub-header title: text-[12px] font-medium — never <h1> or 18px.
❌ Don't
  • Never invent new layout patterns — use defined shells only.
  • Never modify the navigation pattern without approval.
  • Never add page-level tabs unless explicitly requested.
  • Never use "Prevalent AI" text in the topbar — logo image only.

Colors & Tokens

✅ Do
  • Use CSS variables only — var(--shell-accent), var(--shell-text), etc.
  • Spacing on 4pt grid only — 4, 8, 12, 16, 20, 24, 32, 48px.
  • Severity colors: critical #D12329, high #D98B1D, medium #6360D8, low #31A56D.
❌ Don't
  • Never hardcode hex or px values — always use CSS variables.
  • Never use spacing outside the 4pt grid (e.g. 5px, 10px, 15px).
  • Never use shadow-lg, rounded-xl, or other non-system Tailwind utilities.

Buttons

✅ Do
  • All buttons use border-radius: 44px (pill). No exceptions.
  • Max 1 primary button per view.
  • Disabled: opacity: 0.4, cursor-not-allowed — always visible.
  • Destructive actions use red — never purple.
❌ Don't
  • Never use rounded-md, rounded-lg, rounded-xl, or rounded-full.
  • Never show two primary buttons in the same view.
  • Never hide disabled buttons — keep visible at 40% opacity.
  • Never use purple for destructive actions.

Tables & Row Actions

✅ Do
  • Column order: checkbox → data → status → actions. Max 7 columns.
  • Row actions: opacity: 0 → group-hover: opacity-100. Space always reserved.
  • Severity/status always visible in column — never tooltip-only.
  • Cards and table wrappers: border-radius: 4px only.
❌ Don't
  • Never use display:none for row actions — use opacity/visibility.
  • Never show severity only in a tooltip.
  • Never exceed 7 columns in a table.
  • Never use rounded-md or larger on cards.

KPI Cards

✅ Do
  • KPI cards: value + label + delta only. Max 5 per page.
  • Delta uses directional arrow + color based on context (green = good, red = bad).
❌ Don't
  • Never add icons, colored borders, shadows, or custom bg to KPI cards.
  • Never put more than 5 KPI cards on a single page.

Modals & Destructive Actions

✅ Do
  • Button order: Cancel left, Confirm right — always.
  • Destructive modals must name the item and state the consequence.
  • Destructive confirm button uses red/danger variant.
❌ Don't
  • Never use purple for a destructive confirm — always red.
  • Never allow a destructive action without a named confirmation modal.

Charts

✅ Do
  • HTML pages: only the 5 SVG chart functions from charts.md.
  • React: Recharts only — AreaChart, BarChart, LineChart.
  • Always use defined severity colors for data series.
❌ Don't
  • Never use canvas, Chart.js, D3, ECharts, or custom SVG functions.
  • Never use Recharts in HTML pages.
  • Never invent chart types not in the design system.

Personas

✅ Do
  • CISO: KPI cards first (max 5), trend chart, 1 dominant CTA.
  • GRC: Compliance table, control status visible, export button.
  • Security Engineer: Dense table, bulk toolbar, SLA column, pagination.
  • SOC Analyst: Alert queue first, severity sorted, quick row actions on hover.
  • Security Architect: CVSSv3 scores, technical detail, asset context.
❌ Don't
  • Never mix persona patterns — identify one and apply it consistently.
  • Never use a CISO layout for an engineer's workflow or vice versa.
